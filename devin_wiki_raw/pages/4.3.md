## Purpose and Scope

This document explains the plugin registry and asset pipeline system, which orchestrates how plugins process events in Amass. The registry manages handler registration from plugins, constructs priority-ordered pipelines for each asset type, and coordinates execution of handlers across multiple concurrent sessions.

For information about the dispatcher that feeds events into these pipelines, see [Event Dispatcher](#4.1). For details on individual plugin implementations, see [Plugin System](#6).

## Registry Component

The `registry` struct maintains the central index of all registered handlers and constructs asset pipelines on demand.

### Registry Data Structure

| Field | Type | Purpose |
|-------|------|---------|
| `handlers` | `map[string]map[int][]*Handler` | Maps asset type → priority → handlers |
| `pipelines` | `map[AssetType]*AssetPipeline` | Maps asset type → constructed pipeline |
| `logger` | `*slog.Logger` | Logging instance |

The registry organizes handlers in a two-level map structure:
- **First level:** Asset type (e.g., "FQDN", "IPAddress")
- **Second level:** Priority integer (1-9)
- **Value:** Array of handlers at that priority

This structure enables efficient pipeline construction by iterating through priorities in order.

**Sources:** [engine/registry/pipelines.go:19-31]()

## Pipeline Construction

The registry builds pipelines after all plugins have registered their handlers via `BuildPipelines()`:

```mermaid
graph TD
    Start["BuildPipelines()"] --> IterTypes["Iterate asset types<br/>in handlers map"]
    IterTypes --> BuildOne["buildAssetPipeline(atype)"]
    
    BuildOne --> IterPrio["For priority 1 to 9"]
    IterPrio --> CheckHandlers{"handlers[atype][priority]<br/>exists?"}
    CheckHandlers -->|No| NextPrio["Next priority"]
    NextPrio --> IterPrio
    
    CheckHandlers -->|Yes| CountHandlers{"len(handlers) == 1?"}
    
    CountHandlers -->|Yes| CheckMax{"MaxInstances > 0?"}
    CheckMax -->|Yes| FixedPool["pipeline.FixedPool(id, task, max)"]
    CheckMax -->|No| FIFO["pipeline.FIFO(id, task)"]
    
    CountHandlers -->|No| Parallel["pipeline.Parallel(id, tasks...)"]
    
    FixedPool --> AddStage["Append to stages[]"]
    FIFO --> AddStage
    Parallel --> AddStage
    
    AddStage --> NextPrio
    IterPrio -->|Done| CreatePipeline["pipeline.NewPipeline(stages...)"]
    CreatePipeline --> CreateQueue["NewPipelineQueue()"]
    CreateQueue --> StartExec["go ExecuteBuffered(ctx, queue, sink)"]
    StartExec --> Return["Return AssetPipeline"]
```

### Pipeline Stage Types

The registry creates three types of pipeline stages based on handler configuration:

1. **FixedPool Stage** - Single handler with `MaxInstances > 0`
   - Creates a worker pool with fixed concurrency
   - Example: `pipeline.FixedPool("FQDN - Priority: 1", handlerTask(h), 50)`
   - Used for handlers that need concurrency control

2. **FIFO Stage** - Single handler with `MaxInstances = 0`
   - Sequential processing
   - Example: `pipeline.FIFO("FQDN - Priority: 5", handlerTask(h))`
   - Used for handlers that don't need parallelism

3. **Parallel Stage** - Multiple handlers at same priority
   - Executes all handlers concurrently for same event
   - Example: `pipeline.Parallel("FQDN - Priority: 4", task1, task2, task3)`
   - Used when multiple plugins handle same asset type at same priority

**Sources:** [engine/registry/pipelines.go:33-79]()

### Pipeline Execution

Each constructed pipeline runs in its own goroutine:

```go
go func(p *AssetPipeline) {
    if err := p.Pipeline.ExecuteBuffered(context.TODO(), p.Queue, makeSink(), bufsize); err != nil {
        r.logger.Error(fmt.Sprintf("Pipeline terminated: %v", err), "OAM type", atype)
    }
}(ap)
```

The `ExecuteBuffered` method processes events from the `PipelineQueue` through all stages in priority order.

**Sources:** [engine/registry/pipelines.go:73-78]()

The `PipelineQueue` wraps a standard queue and implements the `pipeline.InputSource` interface:

    InnerQueue --> Signal

### PipelineQueue Methods

**`Next(ctx context.Context) bool`** - Blocks until data is available or context is cancelled:
- Returns `true` if queue has data
- Returns `false` if context is done
- Checks every 100ms with ticker
- Also listens on queue's signal channel

**`Data() pipeline.Data`** - Returns next event from queue:
- Dequeues next `EventDataElement`
- Skips events from terminated sessions
- Returns `nil` if queue is empty

**`Error() error`** - Always returns `nil` (no error handling in queue)

## Handler Task Execution

Each handler in a pipeline stage executes via the `handlerTask` wrapper function:

```mermaid
graph TD
    Start["handlerTask(handler)"] --> Check1{"data == nil?"}
    Check1 -->|Yes| ErrNil["Return error"]
    Check1 -->|No| Extract["Extract EventDataElement"]
    
    Extract --> Check2{"Context done?"}
    Check2 -->|Yes| SendQueue["Send to completion queue"]
    Check2 -->|No| Check3{"Session done?"}
    Check3 -->|Yes| SendQueue
    
    Check3 -->|No| GetTransforms["Get transformations<br/>from session config"]
    GetTransforms --> Check4{"len(transformations) > 0?"}
    
    Check4 -->|No| Execute["Execute handler.Callback(event)"]
    Check4 -->|Yes| CheckExclude{"allExcludesPlugin?"}
    
    CheckExclude -->|Yes| Skip["Skip handler"]
    CheckExclude -->|No| CheckMatch{"tosContainPlugin OR<br/>CheckTransformations?"}
    
    CheckMatch -->|Yes| Execute
    CheckMatch -->|No| Skip
    
    Execute --> CheckErr{"Error from callback?"}
    CheckErr -->|Yes| AppendErr["Append to ede.Error"]
    CheckErr -->|No| Return
    AppendErr --> Return["Return data"]
    Skip --> Return
    SendQueue --> ReturnNil["Return nil"]
```

### Transformation Filtering

The handler task applies transformation filtering before executing the callback:

1. **Get transformations** - Retrieves transformations from session config for the event's asset type
2. **Check "all" exclusions** - If transformation is "all → all exclude: [pluginName]", skip this plugin
3. **Check plugin match** - If transformation explicitly lists this plugin, execute
4. **Check transform types** - If handler's `Transforms` field matches transformation's "to" types, execute

This filtering system allows users to control which plugins process which asset types via configuration.

**Sources:** [engine/registry/pipelines.go:93-139]()

## Priority System

Handler priorities determine execution order within each asset pipeline. The priority system ranges from 1 (highest) to 9 (lowest):

### DNS Plugin Priority Assignment

The DNS plugin demonstrates strategic priority assignment:

| Priority | Handler | Rationale |
|----------|---------|-----------|
| 1 | DNS-TXT | Must extract organization IDs first for enrichment |
| 2 | DNS-CNAME | Must resolve aliases before IP lookup |
| 3 | DNS-IP | Resolve to IPs after CNAME chain complete |
| 4 | DNS-Subdomains | Enumerate NS/MX after basic resolution |
| 5 | DNS-Apex | Build hierarchy after subdomain discovery |
| 8 | DNS-Reverse | Reverse lookup after forward resolution complete |

### Priority Guidelines

- **Priority 1-3:** Critical early processing (TXT, CNAME, IP resolution)
- **Priority 4-6:** Secondary discovery (subdomains, company search)
- **Priority 7-9:** Enrichment and follow-up (company data, reverse DNS, service probing)

Handlers at the same priority execute in parallel if from different plugins, or sequentially if from the same plugin.

**Sources:** [engine/plugins/dns/plugin.go:61-161]()

## Concurrency Control

The `MaxInstances` field controls concurrent execution of handlers:

### MaxInstances Values

| Value | Behavior | Pipeline Stage Type | Use Case |
|-------|----------|---------------------|----------|
| 0 | Unlimited | FIFO | Sequential processing |
| > 0 | Fixed pool | FixedPool | Rate-limited operations |

### Example: DNS Handlers

Most DNS handlers use `MaxInstances: support.MaxHandlerInstances` (typically 50):

```go
if err := r.RegisterHandler(&et.Handler{
    Plugin:       d,
    Name:         d.txt.name,
    Priority:     1,
    MaxInstances: support.MaxHandlerInstances,  // = 50
    Transforms:   []string{string(oam.FQDN)},
    EventType:    oam.FQDN,
    Callback:     d.txt.check,
}); err != nil {
    return err
}
```

This creates a `FixedPool` stage that processes up to 50 DNS TXT queries concurrently per FQDN pipeline.

**Sources:** [engine/plugins/dns/plugin.go:151-161](), [engine/registry/pipelines.go:44-54]()

### Queue Size Management

The dispatcher maintains pipeline queue sizes between 100 and 500 items:

- **MinPipelineQueueSize = 100** - Refill trigger threshold
- **MaxPipelineQueueSize = 500** - Maximum items to request per session

When a pipeline queue drops below 100 items, the dispatcher requests `500 / num_sessions` items from each session's queue to distribute load evenly.

## Error Handling

Pipeline execution collects errors in the `EventDataElement`:

```go
if err := r.Callback(ede.Event); err != nil {
    ede.Error = multierror.Append(ede.Error, err)
}
```

The sink function sends completed events (with any accumulated errors) to the completion channel, where the dispatcher logs them:

```go
func (d *dis) completedCallback(data interface{}) {
    ede, ok := data.(*et.EventDataElement)
    if !ok {
        return
    }
    
    if err := ede.Error; err != nil {
        ede.Event.Session.Log().WithGroup("event").With("name", ede.Event.Name).Error(err.Error())
    }
    // increment WorkItemsCompleted
}
```

Errors do not stop pipeline processing; they are logged and the event is marked complete.

**Sources:** [engine/registry/pipelines.go:132-135](), [engine/dispatcher/dispatcher.go:161-176]()

## Summary

The plugin registry and pipeline system provides a flexible, priority-driven architecture for processing assets:

1. **Registration Phase** - Plugins register handlers with priorities and asset types during `Start()`
2. **Construction Phase** - Registry builds pipelines by organizing handlers into priority-ordered stages
3. **Execution Phase** - Events flow through pipeline stages, with each handler potentially generating new events
4. **Coordination Phase** - Dispatcher maintains pipeline queues from session work queues

This architecture enables:
- **Priority-based processing** - Critical handlers (TXT, CNAME) run before secondary ones
- **Concurrent execution** - Multiple handlers process events in parallel via `MaxInstances`
- **Transformation filtering** - Users control which plugins process which asset types
- **Cascading discovery** - Handlers can dispatch new events to trigger additional processing

**Sources:** [engine/registry/pipelines.go:1-183](), [engine/dispatcher/dispatcher.go:1-228](), [engine/types/registry.go:1-97]()
